# 寵美自動化維運（pet-grooming-ops）部署指南

> 流程 3（每日營收報表）+ 流程 4（健康監控）的部署位置。
> 流程 1（預約推播）部署在 `appointment-helper`、流程 2（簽約通知）部署在 `pet-grooming-liff`。

---

## 0. 前置：取得 LINE Channel Access Token + 老闆 userId

### A. Channel Access Token
1. [LINE Developers Console](https://developers.line.biz) → 你的 Messaging API Channel
2. Messaging API 分頁 → 「Channel access token」→ Issue（長期 token）
3. 複製整串（很長）

### B. 老闆 LINE userId（U 開頭 33 字）
**最快方法**：用我們現有的 LIFF 自己簽一筆，從檔案 A 「契約紀錄」分頁 I 欄複製 userId 出來。

或：在 Messaging Channel 開 webhook，老闆傳訊息給官方帳號，從 webhook event 撈 `events[0].source.userId`。

> ⚠️ 老闆的 LINE 必須**已加官方帳號為好友**，push message 才推得進去。

## 1. 部署流程 1（預約推播）：appointment-helper

### 程式碼
`tools/appointment-helper/Code.gs` 已加好：
- `dailyPushTomorrowAppt` + `pushLine` + `testPushLine`
- doGet 加 `?health=1` 端點（給流程 4 監控用）
- **APPT_KEY 已從程式碼移除，改用 ScriptProperties 存**

### 操作步驟（順序很重要）

> ⚠️ **必須先設好 ScriptProperty `APPT_KEY` 真值，再貼新版 Code.gs**。
> 如果順序反過來：貼新版 → 重部署 → 還沒設 APPT_KEY → 線上 doGet 立刻全 auth_failed，前端密碼登入也會壞。

1. 開啟既有的 `appointment-helper` Apps Script 專案（線上）
2. 左下「⚙️ 專案設定」→ 指令碼屬性 → 加 3 個：
   - `APPT_KEY` = (從目前線上 Code.gs 的 `const APPT_KEY = "..."` 複製出來，搬過來)
   - `LINE_TOKEN` = (你的 token)
   - `BOSS_USER_ID` = (老闆 userId)
3. 一般 → 時區設 `(GMT+08:00) Taipei`
4. 回編輯器，全選貼上本地最新版 `Code.gs`（裡面已沒有 APPT_KEY 常數）
5. 儲存（Ctrl+S）
6. 編輯器選函式 `testPushLine` → 執行 → 老闆 LINE 應收到「🧪 [預約小幫手] 測試 ...」
7. 編輯器選函式 `testDoGet` → 執行 → 執行記錄應印出今日預約 JSON（不該是 `auth_failed`）
8. 加觸發條件：左側「⏰ 觸發條件」→ 新增觸發條件
   - 函式：`dailyPushTomorrowAppt`
   - 事件來源：時間驅動
   - 類型：日計時器
   - 時間：**下午 4 點到 5 點**
9. **重新部署 Web App**（doGet 程式碼改了一定要重部署，前端才看得到 `?health=1` + 才會用 ScriptProperty）：
   - 右上「部署」→「管理部署作業」
   - 點既有部署右上鉛筆 → 版本選「新版本」→ 部署
   - 部署 URL 不會變，前端的 APPT_API_URL 不用改

### 驗證
- 用 Brave 無痕視窗打 `{你的 exec URL}?health=1` → 應回 `{"ok":true,"service":"appointment-helper","ts":"..."}`
- 用前端密碼登入照樣能查預約（驗 ScriptProperty 讀得到 APPT_KEY）
- 編輯器選 `dailyPushTomorrowAppt` → 執行 → 老闆 LINE 應收到明日預約清單
- 若明日無預約：執行記錄看到「明日無預約，不推播」，LINE 不會收到

## 3. 部署流程 3 + 4：新建 pet-grooming-ops 專案

### 操作步驟
1. 打開 [Apps Script 首頁](https://script.google.com/) → 新增專案
2. 命名 `pet-grooming-ops`
3. 全選貼上本地 `tools/pet-grooming-ops/Code.gs`
4. 儲存
5. ⚙️ 專案設定 → 指令碼屬性 → 加 4 個：
   - `LINE_TOKEN` = (同上)
   - `BOSS_USER_ID` = (同上)
   - `BOSS_EMAIL` = 老闆收報表的 Email
   - `HEALTH_URL` = **兩個 URL，用換行或逗號分隔**（同時監控 LIFF 跟預約小幫手）：
     ```
     https://script.google.com/macros/s/(LIFF部署ID)/exec?health=1
     https://script.google.com/macros/s/(預約小幫手部署ID)/exec?health=1
     ```
     未來想加更多端點，繼續加新行就好。每個 URL 各自獨立 2 小時冷卻計時。
6. 一般 → 時區設 `(GMT+08:00) Taipei`
7. 第一次執行 `testPushLine` 會跳「需要授權」→ 允許所有 scope（要讀 Sheets、寄信、UrlFetchApp）
8. 執行 `testPushLine` → 老闆 LINE 應收到測試
9. 執行 `testRevenueStats` → 看執行記錄，確認讀到資料、計算對
10. 執行 `testRevenueReport` → 老闆 Email + LINE 都應該收到
11. 執行 `testHealthCheck` → 服務正常時不發訊（log 有寫過程）；可暫時把 `HEALTH_URL` 改成壞網址（例如 `https://example.com/404`）測異常警報

### 順手做：初始化「使用教學」分頁
編輯器選函式 `setupGuideSheet` → 執行一次（會跳 Sheets 寫入授權，第一次允許）→ 檔案 A 自動多一個「使用教學」分頁，員工平板 / 老闆手機開檔案 A 就看得到。

之後想更新內容：
- 改 `buildGuideRows_()` 裡的資料 → 重新部署 Apps Script（不用，trigger/手動執行即生效）→ 編輯器再執行 `setupGuideSheet` 一次
- 函式是 idempotent，重複執行只會清空後重寫，不會留舊資料

### 加觸發條件
左側「⏰ 觸發條件」→ 新增觸發條件，加兩個：

**觸發 A — 每日營收報表**
- 函式：`dailyRevenueReport`
- 事件來源：時間驅動
- 類型：日計時器
- 時間：**晚上 8 點到 9 點**

**觸發 B — 健康監控**
- 函式：`healthCheck`
- 事件來源：時間驅動
- 類型：小時計時器
- 時間：**每小時**（會在每小時的 X:00–X:59 之間隨機跑一次，這對監控來說沒差）

> ⚠️ 這個專案**不需要部署為 Web App**，純 trigger 驅動。所以不要去點「部署」，省事。

## 常見坑

- **LINE 推播 401**：Token 錯或過期 → 重發新 Token，更新 ScriptProperties 的 `LINE_TOKEN`
- **LINE 推播 403 user not found**：老闆沒加官方帳號好友
- **trigger 沒跑**：時區沒設 Asia/Taipei → 你以為的 16:00 其實是 UTC 16:00（台灣時間半夜 12 點）
- **MailApp 寄不出去**：第一次沒授權 → 手動執行一次 `dailyRevenueReport` 觸發授權對話框
- **Apps Script 改 code 後 doPost 沒生效**：忘了「管理部署作業 → 新版本 → 部署」（踩坑筆記第 163 條）
- **MTD 計算對不上**：「服務紀錄」分頁 A 欄不是 Date 物件而是字串時，`d instanceof Date` 為 false 會被跳過。檢查 A 欄資料驗證有沒有設成「日期」格式
