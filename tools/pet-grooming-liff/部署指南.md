# 寵美客戶 LIFF 表單 — 部署指南

> 這份是**照著做**的流程清單。全部照順序走完大約 30-45 分鐘。
> 有任何一步卡住，直接回 Claude 問。

> 🏗️ **架構**：HTML + 後端**都**由 Apps Script 伺服（一份 `Code.gs` + 一份 `index.html` HTML 檔）。
> 不走 GitHub Pages → 密碼類常數全部留在 Apps Script，public repo 完全看不到真值。

---

需要這些帳號 / 資產：
- [ ] LINE 官方帳號（老闆的，已有）
- [ ] LINE Developers Console 登入權限
- [ ] Google 帳號 `mayuolulu@gmail.com`（檔案 A 所有者）

## 第 1 步：在檔案 A 建「契約紀錄」工作表

1. 打開檔案 A（客戶資料那份 Sheets）
2. 底下分頁區 → 右鍵 → 新增工作表 → 命名為 **`契約紀錄`**（一字不差，程式靠這個名字找）
3. 在第 1 列填入這些標題（A-I 欄）：

| A | B | C | D | E | F | G | H | I |
|---|---|---|---|---|---|---|---|---|
| 時間戳 | 客戶電話 | 客戶姓名 | 寵物名 | 服務項目 | 金額 | PDF連結 | 哈希值 | LINE_userId |

4. 順便檢查「客戶資料」工作表的欄位，建議有這些 header（有的話程式會自動對應，沒有就跳過）：
   - `電話`（A 欄，必要）
   - `姓名`、`寵物名`、`品種`、`備註`、`建立時間`、`來源`、`本次服務`、`本次金額`、`LINE_userId`

> 💡 不需要全部都要建，但**電話**欄（A 欄）一定要在。其他欄位可後補，程式會容錯。

---

## 第 2 步：在 Google Drive 建「寵美契約存檔」資料夾

1. 打開 [Google Drive](https://drive.google.com)（用 `mayuolulu@gmail.com`）
2. 左側 → 新增 → 資料夾 → 命名 **`寵美契約存檔`**
3. 進入該資料夾 → 看網址列：
   ```
   https://drive.google.com/drive/folders/1ABC123xyz_LONG_STRING_HERE
   ```
   `/folders/` 後面那串就是**資料夾 ID**，**整段複製起來**。

---

## 第 3 步：建 Apps Script 專案 + 貼兩份程式

1. 打開 [Apps Script](https://script.google.com/) → 新專案
2. 專案命名：`寵美客戶 LIFF 表單`
3. **貼 `Code.gs`**：左邊 `Code.gs` → 全選刪掉 → 貼上 `tools/pet-grooming-liff/Code.gs` 完整內容
4. **新增 HTML 檔**：
   - 左邊檔案列旁的「+」→ 選「**HTML**」→ 命名 **`index`**（不要加 `.html`，Apps Script 會自動處理）
   - 把 `tools/pet-grooming-liff/index.html` 的**完整內容貼進去**
5. 改 `Code.gs` 第 16-27 行的設定：
   - `FILE_A_URL`：已預填寵美檔案 A 網址（若 ID 不同要改）
   - `PDF_FOLDER_ID`：貼上第 2 步拿到的資料夾 ID
   - `LIFF_ID`：**先留佔位符**，第 4 步拿到後再回來填
   - `LIFF_KEY`：改成你自己的亂數密碼（16+ 字元，越亂越好）
   - `SHOP_NAME` / `SHOP_ADDRESS` / `SHOP_TEL`：店家資訊（PDF 抬頭用）
6. Ctrl+S 儲存兩份檔案

> ⚠️ `index.html` 裡的 `LIFF_ID` / `LIFF_KEY` / `LIFF_API_URL` 是 `<?= ... ?>` 模板語法，**不要手改**。執行時 `doGet()` 會從 `Code.gs` 的常數注入。

---

## 第 4 步：部署 Apps Script 為 Web App

1. 右上角「**部署**」→「**新增部署作業**」
2. 齒輪圖示 → 選「**網頁應用程式**」
3. 設定：
   - **說明**：`寵美 LIFF v1`
   - **執行身分**：**我（mayuolulu@gmail.com）**
   - **誰可以存取**：**所有人**（⚠️ 一定要選，不然 LIFF 會被擋）
4. 按「**部署**」→ 同意權限（Sheets / Drive 第一次會彈）
5. 會出現「**網頁應用程式網址**」（`.../exec` 結尾）→ **整串複製起來**

> 🔄 **之後改程式要重新部署時**，選「**管理部署作業**」→ 鉛筆圖示「**編輯**」→ 版本選「**新版本**」→ 部署。這樣網址會保持不變。如果按「新增部署作業」會拿到新網址，LIFF Endpoint URL 要重填。

## 第 5 步：建 LINE LIFF App

1. 登入 [LINE Developers Console](https://developers.line.biz/console/)
2. 選 Provider（沒有就建一個叫「寵美工具」）
3. 進入你的**官方帳號 Channel**（Messaging API 類型的那個）
4. 上方頁籤切到「**LIFF**」
5. 按「**Add**」新增：
   - **LIFF app name**：`寵美資訊登記`
   - **Size**：`Full`
   - **Endpoint URL**：貼第 4 步拿到的 Apps Script `.../exec` 網址（**不用加尾斜線**）
   - **Scope**：勾 `profile` + `openid`（需要讀客戶名字）
   - **Bot link feature**：`On (Aggressive)` 或 `Off` 都可
   - **Scan QR**：保持預設
6. 建好後會拿到一個 **LIFF ID**（格式類似 `2001234567-abCdEfGh`）→ **整串複製起來**

---

## 第 6 步：把 LIFF ID 填回 Apps Script

1. 回到 Apps Script 編輯器 → `Code.gs`
2. 第 21 行的 `LIFF_ID` 改成第 5 步拿到的 ID
3. Ctrl+S 儲存
4. **重新部署**（第 4 步的「管理部署作業 → 編輯 → 新版本」流程）

---

## 第 7 步：先用 testDoPost 驗證後端

1. Apps Script 編輯器上方 **執行** → 選 `testDoPost` 函式 → 點「執行」
2. 第一次會彈權限：Sheets + Drive → **全部授權**
3. 執行完看「執行記錄」→ 應該出現 `{"ok":true,"pdfUrl":"...","hash":"..."}`
4. 驗證：
   - [ ] Drive「寵美契約存檔」有測試 PDF（檔名含 `0912345678_測試寵物_...`）
   - [ ] Sheets「契約紀錄」多一列測試資料
   - [ ] Sheets「客戶資料」多一筆 `0912345678`
5. **測完把測試資料全刪掉**（PDF + 兩個工作表的測試列）

---

## 第 8 步：在 LINE 內測試一筆真實送出

1. 打開 LINE → 加你自己的官方帳號好友
2. 訊息欄貼連結：`https://liff.line.me/<你的_LIFF_ID>` → 送出
3. 點自己送出的訊息中的連結 → 會在 LINE 內開啟 LIFF
4. 填測試資料（姓名、電話、寵物名、勾選項目、打勾同意、簽名）→ 送出
5. 應該會看到「✅ 已送出」畫面 + PDF 連結
6. 驗證：
   - [ ] Sheets「契約紀錄」有新一列
   - [ ] Sheets「客戶資料」有新客戶
   - [ ] Drive「寵美契約存檔」有新 PDF，打開看格式對不對（中文、簽名、注意事項、時間戳、哈希都在）

> 💡 電腦瀏覽器直接打開 `.../exec` 也能看到表單 UI（非 LIFF 環境，`envNote` 會顯示「開發模式」），但這個模式不會真的送資料到後端。

## 🐛 常見問題

**Q1：LIFF 送出時跳「送出失敗：Failed to fetch」**
→ Apps Script 沒部署為「所有人可存取」，或 `LIFF_API_URL` 注入失敗。檢查 `Code.gs` 的 `doGet()` 有沒有貼到、HTML 檔名是不是 `index`。

**Q2：LIFF 一開就白畫面 / 錯誤**
→ LIFF Endpoint URL 跟實際 Apps Script 網址不一致（例如重新部署後忘了更新 LIFF App 的 Endpoint）。回第 5 步編輯 LIFF，貼新網址。

**Q3：打開 LIFF 看到 `<?= LIFF_ID ?>` 原始文字**
→ HTML 檔不是用 `HtmlService.createTemplateFromFile` 載入，或 HTML 檔名不叫 `index`。檢查 `Code.gs` 的 `doGet()` 和 HTML 檔名。

**Q4：送出成功但沒看到 PDF 連結**
→ `PDF_FOLDER_ID` 填錯或資料夾沒權限。去 Apps Script 執行 `testDoPost`，看執行記錄的錯誤。

**Q5：PDF 裡中文變框框**
→ Apps Script 產 PDF 時字體問題，已用 Noto Sans CJK TC fallback。如果還有問題，改成 `PingFang TC` 或用內建 `sans-serif`。

**Q6：換了 `LIFF_KEY`，LIFF 送出就報 auth_failed**
→ 只改 `Code.gs` 就好（因為 `index.html` 是從這邊注入）。改完要**重新部署**才會生效。

## 🔐 安全提醒

- **`Code.gs` 的 `LIFF_KEY` / `LIFF_ID` 是真值，但 `Code.gs` 只存在 Apps Script 線上編輯器**（private）+ 本地 repo（佔位符）。public repo 不會看到真值。
- **`index.html` 用 `<?= LIFF_KEY ?>` 模板注入**，public repo 裡只有模板語法，沒有真值。
- Apps Script 產出的 HTML 在客戶瀏覽器是 runtime 才有真值 → 技術上仍可被 DevTools 看到（這是前端的限制），但**至少不進 GitHub 歷史**。
- 如果不小心把真值 commit 到 public repo → 立刻改 `Code.gs` 的 `LIFF_KEY`、重新部署、把 repo 那行改回佔位符重新 commit。
